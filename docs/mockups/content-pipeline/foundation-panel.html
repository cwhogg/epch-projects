Positioning Statement

v3 edited

April Dunford

For pet owners who worry about their dog's health between vet visits, PetPulse is a continuous health monitoring platform that detects early warning signs of illness through daily behavior analysis and biometric tracking. Unlike general pet fitness trackers (Whistle, Fi) that focus on activity metrics, PetPulse combines veterinary-grade vital sign monitoring with AI pattern recognition to surface clinically meaningful health changes — the kind a vet would want to know about. The competitive alternatives are: - Whistle/Fi: Activity and GPS trackers. They tell you how far your dog walked, not whether something is wrong. - Vet telehealth (Pawp, Airvet): Reactive — you call when you already suspect a problem. No continuous monitoring. - Pet insurance: Financial backstop, not early detection. PetPulse occupies the gap between "my dog seems fine" and "we caught it early." The product's fundamental bet is that continuous passive monitoring catches problems days to weeks before behavioral changes become obvious to owners. Key differentiator: We are NOT a replacement for veterinary care. We are the early warning system that makes vet visits more productive by surfacing data patterns before symptoms are visible. Every communication reinforces: "PetPulse helps your vet help your pet." Target: Dog owners aged 30-55 who spend $100+/month on pet care and have had at least one scare (emergency vet visit, unexpected diagnosis). They are not price-sensitive on pet health but are time-poor and anxiety-prone about their pet's wellbeing. NOT targeting: Cat owners (V2), budget pet owners, breeders/professional handlers, people who want a GPS tracker.

Implementation Critique

Comparing actual implementation (page.tsx) against the design doc spec:

MISSING

Per-card Generate and Edit buttons

Design doc specifies "Buttons: Generate, Edit, View" per card. Implementation only has "View". No way to generate a single document or enter the editor from the panel.

MISSING

Per-card Retry button on error

Design doc specifies individual Retry buttons on failed cards during partial generation failure. Implementation shows "error" text label but no recovery action.

GAP

No timestamp display

Design doc shows "Generated timestamp" and "Last edited timestamp" per card. Implementation has the data (generatedAt, editedAt) but only surfaces editedAt as a boolean "edited" badge. No dates visible.

GAP

"Ready" vs "Empty" not visually distinct

Both un-generated states look identical — no visual indicator that a card can generate vs cannot. The "Requires: X" hint only appears when upstream is missing, but the "ready to generate" state has no affordance. This matters less without per-card Generate buttons, but if those are added, the distinction becomes important.

GOOD

Dependency logic matches design doc

canGenerate() correctly implements the creation hierarchy: Strategy is always ready, Positioning needs Strategy, items 3-6 need Positioning, Social Media also needs Brand Voice, Design Principles needs both Strategy and Positioning.

GOOD

Polling pattern correct

3-second polling interval when progress.status === 'running', stops when generation completes. Matches the existing progress polling conventions.

NOTE

Inline styles instead of design system tokens

Implementation uses inline style={{}} with CSS variable references. This works but bypasses the Tailwind utility class conventions used elsewhere in the codebase. Not a functional problem, but inconsistent with the rest of the app's styling approach.
